设计模式

设计原则

  1. 单一职责原则：一个类仅有一个引起变化的因素（降低耦合性）

  2. 开闭原则：软件实体对扩展开放对修改封闭，及可以在不修改的情况下扩展

    • 通过接口和抽象类规定类的特征：相对稳定不易更改，完成对修改封闭
    • 从抽象类中导出具体类即对扩展开放

  3. 里氏代换原则：超类可被子类替换

    • 反之不成立

  4. 依赖倒转原则：高层模块不依赖与底层模块。应依赖于相应的抽象，即面向接口编程（请举例）

    • 应当使用接口和抽象类进行变量类型声明、参数类型声明、方法返还类 型说明，以及数据类型的转换等，而不要使用具体类。、
    • 传统的过程性系统的设计办法倾向于使高层次的模块依赖于低层次的模 块，抽象层次依赖于具体层次。倒转原则就是把这个错误的依赖关系倒 转过来

  5. 合成复用原则：尽量使用对象组合而不是继承来达到复用的目的

    • 在新对象中使用已有的对象，对已有对象进行委派达到目的
    • 其次考虑继承以遵循前述原则，滥用继承可能会增加系统构架和维护时的复杂度

  6. 接口隔离原则：客户端不应该依赖他不需要的接口

    • 使用多个专门的接口比使用单一的总接口好。依赖建立于最小的接口上
    • 当一个接口过大，应将其分为多个小接口，使用该接口的客户端只需要知道与之相关的方法

  7. 迪米特法则：一个软件实体应当尽可能少的与其他实体发生相互作用

    • 每一个软件单位对其他的单位都只有最少的知识，而且局限于那些 与本单位密切相关的软件单位
    • 当一个模块修改时，就会尽量少的影响其他的模块。扩展会 相对容易。这是对软件实体之间通信的限制。它要求限制软件实体 之间通信的宽度和深度

设计模式概念

代码设计经验总结。描述不断重复事件及其解决方法。提高代码可重用性、可阅读性和可靠性。

四个基本要素：模式名、问题、解决方案、效果

作用：

  1. 重用设计
  2. 方便交流
  3. 方便文档编写
  4. 减少重构代价，确保代码正确性，减少错误
  5. 提供框架
  6. 节省时间

分类：

  1. 目的
    1. 创建型模式：怎样创建（将创建与使用分离）
    2. 结构型模式：如何将类和对象布局
    3. 行为型模式：类或对象之间如何相互配合工作怎样进行智能分配
  2. 范围
    1. 类模式：类与类之间的关系，通过继承来建立，静态的，编译时确定
    2. 对象模式：对象之间的关系，组合或聚合实现，在运行时可以变化，具有动态性

工厂模式

比较：

  1. 简单工厂：
    • 不符合开闭原则，用户不感知具体的类，修改时需要修改工厂类
  2. 工厂模式：

评价：

  1. 优点：
    • 保证客户端只使用同一个产品族的对象
  2. 缺点：
    • 产品的增加困难
    • 增加了系统的抽象性和理解难度

3.策略模式

简化多种算法选择的if-else语句

将算法和对象分开

将一组算法相同的过程提取到抽象类中

逻辑判断在client中

封装了算法

方便算法替换

客户端决定算法的使用

其重心在于如何组织调度算法

算法的平等性

优点：

  1. 提供了一种替代继承的方法，而且既保持了继承的优点（代码重用）还比继承更灵活 （算法独立，可以任意扩展）
  2. 它把采取哪一种算法或采取哪一种行为的逻辑与算法本身分离，避免程序中使用多重 条件转移语句，使系统更灵活，并易于扩展。
  3. 遵守大部分设计原则，高内聚、低偶合

缺点：

  1. 客户端必须知道所有的策略类，并自行决定使用哪一个策略类。这就意味着客户端必 须理解这些算法的区别，以便适时选择恰当的算法类。换言之，策略模式只适用于客 户端知道算法或行为的情况。
  2. 由于策略模式把每个具体的策略实现都单独封装成为类，如果备选的策略很多的话， 那么对象的数目就会很可观。

4.迭代器模式

在客户访问类与聚合类之间插入一个迭代器，这分离了聚合对象与其遍历行为，对客户也隐藏了其内部细节， 且满足“单一职责原则”和“开闭原则”。

5.数据访问对象模式（DAO）

将数据访问从高级业务中分离

内容：

  • 数据访问接口
  • 数据访问实体类
  • 模型对象/数值对象

优点：隔离数据层

缺点：代码量增加

6.MVC模式

组件特征：内容、外观、行为

MVC模式：

  1. 模型：
    • 存储完整的内容
    • 实现改变内容和查找内容的方法
    • 模型没有用户界面，是完全不可见的
  2. 视图
    • 一个模型可以有多个视图
    • 每个视图可以显示全部或不同部分和不同方面的内容
    • 模型更新时，相关联的视图同步更新
  3. 控制器：
    • 使视图和模型分离
    • 负责处理用户输入事件
    • 决定对事件对模型的更改

7.生产者-消费者模式

  • 如果生产者生产速度过快，消费者消费的很慢，并且缓存区达到了最大时。 缓存区会阻塞生产者，让生产者停止生产，等待消费者消费了数据后，再 唤醒生产者
  • 当消费者消费速度过快时，缓存区为空时。缓存区则会阻塞消费者，待生 产者向队列添加数据后，再唤醒消费者。
  • 数据一致问题：同步锁（synchronized）
  • 消费者和生产者的协同：线程内部通信（wait（），notify（））

缓冲区：

缓冲区的判断条件为while不是if：防止错误的唤醒

java 中wait（）在同步块内： 防止出现Lost Wake-up

优点：

  1. 并发：通过异步实现高并发，将一个耗时的流程进行拆分

  2. 解耦：生产和消费进行解耦

8.观察者模式

一个对象的状态或行为的变化将导致其他对象的状态和行为也发生改变

  • 定义了对象间一种一对多的依赖关系
  • 发生改变的对象称为观察目标，被通知的对象为观察者
  • 一个观察目标可以对应多个观察者
  • 是一种对象行为型模式

优点：

  1. 实现表示层和数据逻辑的分离
  2. 观测目标和观察者之间的抽象的耦合
  3. 支持广播通信，简化了一对多系设计的难度
  4. 符合开闭原则，新增具体的观察者无需修改

缺点：

  1. 遍历观察者开销大
  2. 存在循环依赖会导致系统崩溃
  3. 没有响应的机制让观察者知道所观察的对象发生变化的原因

使用场景：

  1. 一个抽象模型有两个方面，其中一个方面依赖于另一个方面，将这两个方面 封装在独立的对象中使它们可以各自独立地改变和复用
  2. 一个对象的改变将导致一个或多个其他对象发生改变，且并不知道具体有多 少对象将发生改变，也不知道这些对象是谁
  3. 需要在系统中创建一个触发链
